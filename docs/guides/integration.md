# Integration Patterns Guide

This guide explores patterns and best practices for integrating various GraphAI Utils packages with each other and with external services to build robust AI applications.

## Core Principle: Decoupling and Composition

GraphAI Utils are designed as a suite of focused packages. Effective integration often involves:

1.  **Decoupling**: Each part of your system (API, visualization, specific AI logic) should be as independent as possible.
2.  **Composition**: Combine these decoupled parts to build complex applications. GraphAI workflows themselves are a form of composition.

## Common Integration Scenarios

### 1. Backend API + Frontend Visualization

This is a very common pattern:
-   **Backend**: `@receptron/graphai_express` or `@receptron/graphai_firebase_functions` serves GraphAI workflows as an API.
-   **Frontend**: A Vue.js (`@receptron/graphai_vue_cytoscape`) or React (`@receptron/graphai_react_cytoscape`) application consumes this API to:
    -   Fetch graph structures for display.
    -   Trigger graph executions.
    -   Display results or real-time updates.

**Integration Points:**

-   **API Contract**: Define a clear API contract between frontend and backend.
    -   Endpoint for fetching graph data (nodes, edges, layout hints).
    -   Endpoint for running a graph (e.g., OpenAI-compatible `/v1/chat/completions`).
    -   Endpoint for streaming results (if applicable).
-   **Data Transformation**:
    -   Backend's `transformRequest` (in `graphai_express` / `graphai_firebase`) adapts incoming HTTP requests to what the GraphAI workflow expects.
    -   Backend's `transformResponse` adapts GraphAI output for the HTTP response.
    -   Frontend might need to transform API responses into the `elements` format expected by Cytoscape components.
-   **Authentication & Authorization**: Secure API endpoints. The frontend sends credentials (e.g., JWT, API key) which the backend validates.

**Example Flow:**

1.  Frontend loads, calls `GET /api/workflow/{workflowId}`.
2.  Backend (GraphAI Express/Firebase) retrieves the workflow definition (e.g., from a file or database), potentially redacts sensitive parts, and returns it in a format suitable for visualization (nodes, edges).
3.  Frontend uses `@receptron/graphai_vue_cytoscape` or `react_cytoscape` to render the graph.
4.  User interacts with the frontend, triggering an action (e.g., submitting a form).
5.  Frontend calls `POST /api/workflow/{workflowId}/run` (or an OpenAI-compatible path) with payload.
6.  Backend executes the corresponding GraphAI workflow.
7.  Backend returns the result (or streams it).
8.  Frontend updates the UI, possibly highlighting nodes in the graph as they execute or complete.

### 2. Using `@receptron/event_agent_generator`

The agent generator is a Vue-based tool. Its output (GraphAI agent JSON configuration) can be integrated in several ways:

-   **Manual Copy-Paste**: User generates agent JSON and manually pastes it into a larger GraphAI workflow configuration file or a database.
-   **API Integration**:
    -   The Agent Generator UI could `POST` the generated agent JSON to a backend endpoint.
    -   This backend endpoint saves the agent configuration (e.g., to a "custom agents" collection in a database).
    -   Other parts of the system (e.g., a workflow editor dashboard) can then load and use these custom agent configurations.
-   **Dynamic Loading in Workflows**: A "master" GraphAI workflow could have a node that fetches an agent configuration (generated by the tool and stored) from a database or API, and then uses it in a `nestedGraph` or dynamically constructs a part of its workflow.

### 3. Integrating `@receptron/stream_utils`

`stream_utils` are for handling streaming data.

-   **Server-Side (within GraphAI Express/Firebase or Custom Agents)**:
    -   **Parsing Incoming Streams**: If your server receives a stream (e.g., file upload, external data feed), `stream_utils` can parse or transform it before it's fed into a GraphAI workflow. This might happen in `transformRequest` or a dedicated ingestion endpoint.
    -   **Transforming Outgoing Streams**: If a GraphAI agent (like `openAIStreamAgent`) outputs a raw stream, a subsequent custom agent (using `stream_utils`) or the `streamResponseHandler` in `graphai_express` can format this stream (e.g., into Server-Sent Events, line-delimited JSON) before sending it to the client.
    -   **Custom Streaming Agents**: Build agents that internally use `stream_utils` to process or generate streams.
-   **Client-Side**:
    -   If the server sends a custom stream format, the client can use `stream_utils` to parse it. For example, parsing NDJSON or a proprietary binary stream format.

### 4. Firebase Ecosystem Integration (`@receptron/firebase-tools`)

`@receptron/firebase-tools`, especially `firebaseStreamFilter.ts`, suggests integration with Firebase event sources that produce streams.

-   **Firestore Change Streams**: A Firebase Function triggered by Firestore changes could receive a stream of document changes. `firebaseStreamFilter` might adapt this stream into a format more easily consumable by a GraphAI workflow.
-   **Cloud Storage Events**: Processing large files uploaded to Firebase Storage. A function triggered by `onFinalize` could get a file stream, which `firebaseStreamFilter` might process (e.g., parse CSV lines, extract image frames) before feeding to GraphAI.
-   **Pub/Sub Triggers**: Messages from Pub/Sub could be processed as a stream.

**Conceptual Flow with Firestore Stream:**
1.  Data changes in Firestore.
2.  Firebase Function (`onWrite` trigger) is invoked.
3.  The function receives change data (potentially as a stream or batched).
4.  If it's a stream, `firebaseStreamFilter` (from `@receptron/firebase-tools`) processes/adapts it.
5.  The adapted data/stream is passed as input to a GraphAI workflow executed by `@receptron/graphai_firebase_functions`.
6.  GraphAI processes the data.
7.  Results might be written back to Firestore or another service.

### 5. Integrating Multiple GraphAI Workflows

Applications often consist of multiple, specialized GraphAI workflows.

-   **Chaining Workflows via API Calls**:
    -   Workflow A (e.g., data ingestion) runs and completes.
    -   Its output is then used as input to trigger Workflow B (e.g., analysis) via an HTTP call from the client or another backend service.
-   **`nestedGraph` Node**: One GraphAI workflow can invoke another as a sub-process using a `nestedGraph` property within a node. This allows for modular workflow design.
    ```typescript
    // const mainWorkflow: GraphAIConfig = {
    //   nodes: {
    //     preprocess: { agent: "dataCleanerAgent", inputs: [":$0"] },
    //     complexAnalysis: {
    //       agent: "graphExecutorAgent", // A conceptual agent that runs another graph
    //       // or direct support for nestedGraph:
    //       nestedGraph: subWorkflowConfig, // The config for the sub-workflow
    //       inputs: [":preprocess"] // Pass data to the sub-workflow
    //     }
    //   }
    // };
    ```
    GraphAI core supports `graphData` and `agent` properties on nodes which can be functions. A node's agent could be a function that itself instantiates and runs another `GraphAI` instance.

-   **Message Queues (e.g., RabbitMQ, Kafka, Google Pub/Sub)**:
    -   Workflow A publishes its result to a message queue topic.
    -   A listener (e.g., another Firebase Function, a separate microservice) consumes messages from this topic and uses them as input to trigger Workflow B. This is excellent for decoupling and asynchronous processing.

## Data Flow and Transformation

-   **Consistency**: Aim for consistent data structures between your GraphAI workflows and the services/UI consuming them.
-   **Adapter Pattern**: Use `transformRequest` / `transformResponse` in API layers, or dedicated "adapter" agents within GraphAI, to map data between different formats.
-   **Validation**: Validate data at the boundaries:
    -   When receiving HTTP requests.
    -   When receiving inputs in custom agents.
    -   Before sending data to external services.

## Security Considerations in Integration

-   **Authentication**: Secure all API endpoints. Use API keys, OAuth, or Firebase Auth.
-   **Authorization**: Ensure a caller (user or service) has permission to execute a specific workflow or access certain data.
-   **Input Sanitization/Validation**: Especially if user-provided data is passed into `params` of agents that might interpret strings as code or commands (though well-designed agents avoid this).
-   **Secrets Management**: Store API keys and sensitive configuration securely (e.g., Firebase Functions config, environment variables, HashiCorp Vault). Do not hardcode them.
-   **Rate Limiting and Quotas**: Protect your backend services from abuse.

This diagram illustrates how different components can be integrated to form a complete application. The key is the API layer that decouples the frontend from the specific GraphAI backend implementation.

By understanding these patterns, you can effectively combine the strengths of each GraphAI Utils package to build sophisticated and scalable AI applications.
